### ② 멘트
> 이 페이지는 **수정 후 최종 결과를 한 화면에 모은 것**. 6개 이벤트 카드를 봐.
>
> 색깔이 회색이면 비유의(Non-Sig), 노랑이면 Marginal, 초록이면 Safe Haven. 보면 **Safe Haven 카드가 하나도 없어**. 5개는 Marginal, 1개(호르무즈)는 Non-Sig. 즉 **±10일 윈도우에서 보면 BTC 가 일관된 안전자산이라고 분류되는 이벤트가 0개**.
>
> 이게 우리 분석의 결론이고, 빨간 박스에 한 줄로 정리해뒀어.
>
> 카드의 각 수치 의미를 짚으면 — CAR 은 이벤트 ±10일 동안의 누적 초과수익률, p_BMP 와 p_block 은 두 가지 검정의 p-value. 둘 다 0.05 미만이어야 Safe Haven 으로 분류돼. 보면 어느 카드도 두 p 가 동시에 0.05 밑이 안 돼.
>
> "근데 이건 ±10일 윈도우만 본 거잖아? 다른 윈도우에서는 다를 수 있지 않아?" 그게 다음 페이지에서 우리가 검증한 거야. 윈도우를 4개로 늘려서 결과가 robust 한지.

### ③ 수정 근거
- **C1 수정 결과 첫 번째 화면**. 팀원 원본은 ±17 또는 ±3 단일 윈도우만 사용. 우리는 4개 윈도우(±3/5/10/17) 동시 분석으로 robust 한 결과만 카드에 표시. ±10 을 대표 윈도우로 채택한 이유는 추정기간(95~120일) 대비 적정 비율이고 BMP 검정의 표본 효율 균형이 좋아서.

## 📍 [3:30–6:30] 페이지 2: 🔬 Event Study (윈도우 sensitivity)

### ① 화면에서 보이는 수치
자산 = **BTC** 선택 후:
- **CAR 히트맵** (윈도우 ±3/5/10/17 × 6 이벤트)
- **분류 결과 표** (Safe / Risky / Marginal / Non-Sig)
- **p-value 표** (BMP 와 Block 중 작은 값을 색상 그라데이션으로)
- **Robustness 진단 표**

핵심 수치:
- 이스라엘-하마스 BTC CAR: **±3 → −0.039**, ±5 → +0.009, ±10 → +0.209, **±17 → +0.328**
- 분류: ±3 Non-Sig → ±5 Non-Sig → ±10 Marginal → **±17 Safe Haven (p=0.012)**
- Robustness 표: 이-하마스만 **⚠️ 불안정** (분류 종류 3/4), 나머지 5개 ✅ 안정

### ② 멘트
> 이 페이지는 가장 중요한 진단이 들어 있어.
>
> 우리 원본 노트북은 처음에 EVENT_WINDOW = 17 로 시작했고, 그 다음 셀에서 = 3 으로 다시 돌렸어. 그런데 둘 다 결과가 다른데, 어느 게 진짜인지 검증한 적이 없어. 그래서 내가 ±3, ±5, ±10, ±17 4개 윈도우로 똑같은 분석을 동시에 돌려서 비교했어.
>
> 히트맵 가운데 행 — **이스라엘-하마스** 봐줘. ±3일 CAR 은 −0.039, **음수**야. ±5 일은 +0.009 거의 0. ±10 은 +0.209, ±17 일에서 비로소 +0.328 로 가장 커져. 분류 표를 봐도 같은 패턴 — ±3, ±5 는 Non-Sig 인데 ±17 에서만 Safe Haven 으로 분류돼.
>
> 즉 **"이스라엘-하마스 = Safe Haven" 이라는 팀원 원본의 결론은 ±17 일을 선택했기 때문에 나온 결과**야. 윈도우를 짧게 잡으면 효과가 사라져.
>
> 다른 5개 이벤트는 robustness 진단 표에서 다 ✅ 안정으로 떠. 즉 어떤 윈도우를 잡든 결론이 거의 같아 — Non-Sig 또는 Marginal. 이 5개에서 BTC 는 안전자산이라는 증거가 없어.
>
> **결론**: 이스라엘-하마스 사례는 윈도우 의존성이 강해서 robust 한 증거로 쓰기 어려워. 따라서 BTC 안전자산 가설을 뒷받침할 이벤트가 6개 중 0개야.

### ③ 수정 근거
- **C1 수정**: 팀원 원본 셀 #03 (`EVENT_WINDOW = 17`) 과 셀 #21 (`EVENT_WINDOW = 3`) 사이에서 결과가 다른데 둘 중 어느 게 옳은지 검증 없이 두 결과를 모두 발표.
- 학술 표준 — robustness 검증을 위해 단일 윈도우가 아니라 **여러 윈도우 sensitivity 분석** 필요. 보통 Event Study 표준은 ±5, ±10, ±20 비교.
- 우리는 표본 한계 (이벤트별 34~38 거래일) 때문에 ±17 이 상한이라 ±3, ±5, ±10, ±17 4개로 설정.

### ④ 자산 셀렉터로 다른 자산도 보여주기 (옵션, 시간 있으면)
> Gold 로 바꿔보면 — 호르무즈 위기에서 ±10 일 CAR=+0.05 정도, 그것도 그냥 Non-Sig. 즉 우리 데이터에서 6 이벤트 ±n 거래일 만으로는 어떤 자산도 명확한 Safe Haven 으로 분류되지 않아. 이건 표본이 작은 게 한계.

## 📍 [6:30–10:30] 페이지 3: 📊 GARCH 변동성 (수정판)

### ① 화면에서 보이는 수치

**[방법 A] arch 라이브러리** 선택 시:
- 모델 비교 표 1순위: A2(GPR_custom) AIC=**1089.07**
- 모든 모델 **beta = 0.0** (라이브러리도 동일)
- alpha 0.14~0.15 (정상 범위)
- γ 계수 표: 모두 비유의 (가장 작은 p 가 A2 GPR_custom = 0.118)

**[방법 B] 커스텀 MLE FIXED** 선택 시:
- 1순위: B2(GPR_custom) AIC=**1087.02**
- B3(VIX+F&G): VIX γ=**+3.50, p=0.040 ✱**
- **B4(공식+심리): VIX γ=+7.56, p=0.001 ✱✱✱**, 공식 GPR γ=−4.91, p=0.052
- alpha 0.13~0.18, omega 12.66~13.41 — 모두 정상 (경계 수렴 ✅ 해소)
- γ 막대그래프 (95% CI 에러바 포함)
- 초록 박스: 새로운 발견 강조

### ② 멘트
> 여기가 우리 수정에서 **가장 큰 변화가 일어난 부분**이야. 두 가지 방법으로 나눠 봤어.
>
> **[방법 A] 먼저 — arch 라이브러리.**
> 우리 원본 노트북은 `from arch import arch_model` 이렇게 import 만 해놓고 실제로는 안 써. 노트북 마크다운에 "arch 가 외생변수 미지원이라 직접 MLE" 라고 적혀 있어. 이게 절반은 맞고 절반은 틀려. arch 라이브러리는 분산방정식 GARCH-X 는 진짜 미지원이지만, **평균방정식**에는 X 넣을 수 있어 (HARX 모델). 즉 우리는 옵션 하나를 모르고 그냥 직접 짠 거야.
>
> 그래서 arch 라이브러리로 평균방정식에 외생변수 넣고 표준 GARCH(1,1)-Student't 돌렸어. 결과는 모델 비교 표 1순위가 GPR_custom 단독 모델, AIC=1089.07. 근데 보면 모든 모델에서 **beta=0** 으로 수렴해. 이건 라이브러리도 마찬가지. 표본 210거래일이 GARCH 권장 300+ 보다 작아서 변동성 지속성을 안정적으로 추정하기 어려운 거야. 외생변수도 다 비유의.
>
> **이게 라이브러리 표준 결과의 한계**. 그럼 팀원 원본 모델(분산방정식 X) 자체가 잘못된 거냐? 그건 아니야. 분산방정식 X 로 봐야 우리 가설("지정학 리스크가 BTC 변동성에 직접 영향") 을 진짜 검증하는 거니까.
>
> **그래서 [방법 B] — 팀원 모델 유지하되 결함만 수정.**
> 라디오 버튼 [B] 로 바꿔봐. 여기가 핵심.
>
> 팀원 원본의 두 가지 critical 결함:
> - **B2**: alpha 가 1e-6 하한에 경계 수렴. GARCH 의 기본 가정이 깨진 상태. t-stat 5000 같은 무의미 수치 나옴.
> - **B3**: omega 가 0 으로 수렴. 분산이 음수가 될 수 있는 정의 위배.
>
> 내가 한 수정 3가지:
> 1. alpha 하한을 1e-6 → **1e-3** 으로 강제. 이러면 alpha=0 같은 값에 도달 못 해.
> 2. **다중 초기값 grid search 8회**. local optimum 함정 회피.
> 3. omega 하한을 `var_y * 0.01` 로 강제.
>
> 결과 — 페이지 아래 "✅ 경계 수렴 해소 검증" 표 봐. **모든 모델에서 alpha 0.13~0.18, omega 12~13** 으로 정상 추정됐어. ✅ 정상 ✅ 정상 다 떠. 경계 수렴 해소 완료.
>
> 그리고 **여기서 진짜 중요한 게 나와**.
>
> 페이지 아래 초록 박스 — **VIX (시장공포지수) 효과**.
>
> - 팀원 원본: 모든 외생변수 비유의 (VIX p=0.82). 결론 "어떤 변수도 BTC 변동성 설명 못 함".
> - 수정 후: **B3 모델에서 VIX γ=+3.50, p=0.040 ✱**. **B4 모델에서 VIX γ=+7.56, p=0.001 ✱✱✱** 매우 유의.
>
> 즉 시장공포가 BTC 변동성에 **양의 강한 영향**을 미치고 있었는데, 팀원 원본의 alpha=0 경계 수렴 때문에 그 효과가 묻혔던 거야.
>
> **해석**: 위기 뉴스 자체가 직접 BTC 변동성을 키우는 게 아니라, **그 뉴스 때문에 시장 전체가 공포에 떨면 BTC 도 따라 흔들린다**. 채널이 GPR → BTC 가 아니라 **GPR → VIX → BTC** 인 거지. 이게 우리 수정에서 나온 가장 큰 발견.
>
> 한 가지 더 — B4 모델 공식 GPR γ=−4.91, p=0.052. **borderline 음의 영향**. 시장공포를 통제하면 공식 GPR 은 오히려 BTC 변동성을 낮추는 방향. 직관 반대인데 해석은 "공식 GPR 은 글로벌 지정학을 너무 뭉뚱그려 측정해서 BTC 신호가 희석된다" 정도.
>
> 막대그래프에서 빨간 막대가 p<0.05 유의 변수, 노란 막대가 p<0.10 borderline. B4 모델 VIX 가 가장 큰 빨간 막대. 95% CI 에러바도 0 을 안 걸치고 양수 영역에 있어.

### ③ 수정 근거
- **B1**: 노트북 셀 #02 의 import 와 셀 #10 의 실제 코드가 모순. arch 라이브러리 docs 확인 결과 평균방정식 X 는 HARX 로 가능. 직접 MLE 가 잘못은 아니지만 SE 부정확.
- **B2 / B3**: 팀원이 자인한 한계 (셀 #11 마크다운: "alpha=0 경계 수렴", "omega=0 수렴") — 본인이 이미 인지. 우리는 그걸 직접 고친 것.
- **B2 fix 정당화**: 금융 시계열에서 alpha 가 정상이면 보통 0.05~0.20 범위. 1e-3 하한은 정상 범위 진입의 최소 조건.
- **B3 fix 정당화**: omega ≥ 0 은 GARCH 정의상 필수. var_y × 0.01 하한은 데이터 분산의 1% 라는 보수적 기저값.

### ② 멘트
> 마지막 critical 수정. 가장 미묘하면서 가장 광범위한 영향을 미칠 수 있는 거.
>
> `master_data_결측치제거_v1.ipynb` 셀 #05 보면, BTC 수익률을 이렇게 처리해놨어. 비트코인은 24시간 거래되니까 토·일에도 가격이 움직이거든. 우리 분석은 NYSE 거래일 기준이니까 토·일 수익률을 어떻게 다룰지가 문제.
>
> 팀원 로직: **월요일 BTC 수익률 = (금 + 토 + 일) / 3 평균합산**.
>
> 이게 비표준이야. 금융 표준은 그냥 **NYSE 거래일 종가차분** — `log(P_월/P_금)`. 비거래일은 무시 (BTC 가 24h 라도 NYSE 와 맞춰야 하니까).
>
> 이 차이가 결과에 얼마나 영향 주는지 정량화하려고 yfinance 에서 BTC 가격 다시 받아서 표준 방식으로 다시 계산했어.
>
> 결과 — 분포 통계 표 봐.
>
> **팀원 BTC 연환산 변동성 53.13% vs 표준 62.86%**. 팀원 로직이 BTC 변동성을 **15.5% 축소**시키고 있었어. F-검정 p < 0.0001 — 이 차이는 통계적으로 매우 유의.
>
> 첨도도 다르게 나와. 팀원 23.58 vs 표준 13.91. 팀원 로직이 평균합산하면서 극단값들을 평탄화시키니까 분포 모양 자체가 달라져.
>
> 두 히스토그램이 위에 그려져 있어 — 주황(팀원) 이 파랑(표준) 보다 **중앙에 몰려 있고 양쪽 꼬리가 작아** 보이지. 이게 변동성 축소 효과의 시각적 증거.
>
> 그리고 자기상관 — Ljung-Box 검정 결과: 표준 BTC 는 lag 10 에서 자기상관이 유의 (p=0.017) 인데 팀원 BTC 는 평탄화돼서 자기상관이 사라짐 (p=0.111). 즉 팀원 로직은 BTC 의 시계열 구조 자체를 왜곡한 거야.
>
> **이게 의미하는 건** — GARCH 의 alpha (충격 효과), beta (지속성), 모든 분위수 회귀, Event Study CAR — **이 모든 분석이 변동성을 사실보다 작게 본 채로 돌아갔어**. 추정량들이 다 편향됐을 가능성이 있어.
>
> 이번 단계에서는 영향도 평가까지만 했고, 표준 BTC 로 master_data 재생성 후 전체 파이프라인 재실행은 못 했어. 근데 그게 다음 단계 1순위야. 결론이 또 바뀔 수 있어.

### ③ 수정 근거
- **A1**: master_clean 셀 #05 의 코드는 표준 금융 시계열 처리법과 다름. Bitcoin 24h 시장과 NYSE 거래일 매칭 시 통상 처리: ① 거래일 종가만 차분 (우리 표준), 또는 ② raw 24h 차분 후 weekly 집계. 평균합산은 비표준.
- **F-검정 정당화**: 분산 동일성 검정. p < 0.0001 → 두 시계열의 분산이 통계적으로 다르다는 강한 증거.
- **Ljung-Box 정당화**: 표준 BTC 의 자기상관이 유의한데 팀원 BTC 는 비유의 → 평균합산이 시계열 의존성을 평탄화시킨 것.

### ① 화면에서 보이는 수치

### ② 멘트
> 마지막 페이지는 우리 모델·수식 정리. 발표 자리에서 "정확히 무슨 모델 썼어?" 질문 나오면 여기 보여주면 돼.
>
> 짧게 정리하면:
>
> **Event Study**: 정상 수익률 추정 → AR = 실제 − 정상 → CAR 누적합 → BMP + Block Bootstrap 검정.
> 모델: BTC 는 NASDAQ 벤치마크 Market Model, Gold/DXY 는 CMRM(평균), TLT/NASDAQ 은 SP500 벤치마크.
> 추정기간 [-120, -26] 95~120일, 이벤트 윈도우 ±3/5/10/17.
>
> **GARCH-X**: 두 방법 — 평균방정식 X (arch lib) vs 분산방정식 X (커스텀 FIXED). t-분포 잔차.
> 우리 가설("지정학 리스크 → BTC 변동성") 에 직접 맞는 건 분산방정식 X 쪽 (방법 B).
>
> **Quantile Regression**: 팀원 원본 유지. Q_τ(BTC) = α + β·SP500_z + γ·GPR_z, τ ∈ [0.05~0.95].
> Safe Haven 조건: τ ≤ 0.10 에서 β < 0 + p < 0.05.
>
> **다중 검정 보정**: 우리가 한 검정 수가 6 이벤트 × 5 자산 × 4 윈도우 = **120건**. Bonferroni α* = 0.000417. **현재 미적용**. 다음 작업 (C3).
>
> 페이지 끝.
